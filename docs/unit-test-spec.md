单元测试强制规范（AI / 开发通用）
1. 代码文件结构约束
所有单元测试统一放置在当前代码文件最底部。
固定固定格式：必须使用 #[cfg(test)] mod tests 包裹，不分散、不新建额外测试文件（单文件内聚）。
测试模块内部禁止嵌套多级子模块，保持扁平结构。
2. 必测范围（强制全覆盖）
所有公开函数：pub fn / 公开结构体方法、对外暴露接口，必须编写单元测试。
核心纯逻辑函数：非工具封装、业务核心计算、解析、转换、校验类私有核心函数，必须补写测试。
新增功能、迭代改造、修复 bug，必须同步新增 / 修改对应单元测试，禁止只改业务不更测试。
3. 测试场景覆盖要求（每条强制）
每个被测函数，必须覆盖三类场景，缺一不可：
正常场景：常规入参、标准流程、预期正确返回。
边界场景：空值、空字符串、极限长度、最小 / 最大阈值、空集合、特殊边界入参。
异常场景：非法入参、错误格式、不存在资源、权限不足、参数缺失、非法枚举 / 状态。
4. 断言使用规范
统一使用 Rust 标准库原生断言：
相等判断：assert_eq!
不等判断：assert_ne!
布尔 / 存在性判断：assert!
禁止使用第三方断言库、自定义宏、非标准断言方法。
关键异常用例，必须断言错误类型 / 错误码 / 返回 Err，不允许只判断 ok。
5. 无依赖隔离规范（纯单元测试）
单元测试零外部依赖：禁止连接数据库、网络请求、调用第三方 API、启动服务、启动 TUI / 渲染进程。
文件 IO、路径操作类逻辑：必须使用临时目录 / 临时文件隔离，不操作本地真实业务路径。
环境变量、系统配置：测试内做隔离重置，不依赖本地开发环境。
外部交互逻辑做分层解耦：核心逻辑抽无副作用纯函数，仅测试纯逻辑层。
6. 测试函数命名规范
统一命名格式：test_功能点_场景描述示例：
test_args_parse_normal_valid 正常场景
test_code_gen_empty_prompt 边界场景
test_file_write_invalid_path 异常场景
命名语义清晰，见名知意；
单个测试函数单一职责，一个用例只测一个逻辑点，禁止多场景混合堆砌。
7. 代码风格与可运行约束
业务代码 + 测试代码，严格遵循项目统一 rustfmt、clippy 规范，无编译警告。
执行 cargo test 可直接完整运行，无编译报错、无用例失败、无跳过硬编码。
测试代码简洁轻量化，不写冗余逻辑、不复制大段重复代码，可适度抽测试共用常量。
8. 附加：CLI 项目专项补充（适配你 Claude Code 类 Rust CLI）
命令行参数解析、子命令、参数校验，必须单独编写测试用例。
退出码、错误提示文案、参数缺失 / 非法参数，纳入异常场景测试。
禁止在单元测试中调用 std::process::exit，统一通过返回值 / 错误码做断言。
日志、彩色输出、终端交互逻辑，测试中默认关闭，避免干扰断言。
